# Retire the `autopilot` plugin (ai-dev/ai-review/ai-qa) ‚Üí CI-native AI review that **gates heavy CI** in `EdulyCom/github-actions`

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:subagent-driven-development` (chosen execution mode ‚Äî a fresh subagent per task with a review gate between tasks). Steps use `- [ ]` for tracking.
> **First implementation action (Task 1.1):** create the public repo `EdulyCom/github-actions`, then **file this plan as a tracking issue in that repo** (`gh issue create --repo EdulyCom/github-actions`), and commit this spec into `EdulyCom/github-actions:docs/` (docs-as-you-go). This plan file is the source spec; the tracking issue mirrors it and each Phase/Task checklist item is checked off there as it lands.

**Goal:** Replace the locally-run `/ai-review` and `/ai-qa` autopilot skills (published via the `mtm-bot-server` webhook) with three reusable **composite GitHub Actions** in a new shared repo `EdulyCom/github-actions` that run the AI review automatically in CI as the **first job**, post a native PR review + labels, **gate heavy CI on the review's `verdict` job output** (heavy lanes `needs:` it ‚Äî no downstream compute until the review passes), and retire the entire `autopilot` plugin (including `ai-dev`) + the webhook server.

**Architecture:** Three composite actions (`auto-assign`, `ai-review`, `ai-qa`) live one-dir-per-action in `EdulyCom/github-actions`, consumed at `@main` (first-party); the third-party actions used *inside* them are SHA-pinned (D13). Each consuming repo copies a thin (~30-line) caller workflow owning triggers/permissions/concurrency. `ai-review` runs the review agent via `anthropics/claude-code-action` (Haiku context stage ‚Üí Sonnet/Opus review stage against a vendored rubric ‚Üí JSON verdict), then a **deterministic `github-script` step** posts a native PR review (APPROVE / REQUEST_CHANGES) + reconciles `‚úì/‚úó /ai-review` labels **and exposes the verdict as a composite-action job output**. The consumer runs this action as the **first CI job** and gates heavy lanes on it (a small `review-gate` job `needs: [ai-review]` fails red unless `verdict == 'pass'`; heavy lanes `needs: [review-gate]`) ‚Äî so a FAIL review withholds all downstream compute. **No App-authored identity-pinned Check Run** (that machinery + the #3414 workaround are dropped); the gate is an ordinary `pull_request` job status, not a label cascade. Claude produces content+verdict only; the workflow posts + gates deterministically.

**Tech Stack:** GitHub Actions (composite), `anthropics/claude-code-action`, `actions/create-github-app-token` (App token used to author labels/review), `actions/github-script`, `actions/checkout`, `gh` CLI, Anthropic API (Haiku/Sonnet/Opus). All third-party actions **SHA-pinned** + Dependabot-bumped (D13); self-CI = `actionlint` + `zizmor` (D14).

## Context ‚Äî why this change

Today `ai-review`/`ai-qa` are autopilot **skills** a developer runs in a local Claude session; they post a marker comment that `EdulyCom/mtm-bot-server` (a NestJS GitHub App webhook server) reads to publish an enforcing Check Run + labels. The gate only runs when a human invokes a skill, and depends on a bespoke always-on server. This change makes the review **run automatically in CI on every PR**, adoptable by **any org's repo** via a thin caller (the actions are org-agnostic; eduly is simply the first consumer), and drops the webhook server. The repo is **public**, so all defaults are generic and eduly-specific values live only in eduly's caller.

**Gate-mechanism decision:** the AI review **gates heavy CI**, but through an **ordinary in-workflow job dependency**, not an App-authored identity-pinned Check Run. The shared action exposes its `verdict` as a **job output**; eduly runs the action as the first CI job, and a small `review-gate` job (`needs: [ai-review]`) fails red unless `verdict == 'pass'`, with every heavy lane `needs: [review-gate]` ‚Äî so a FAIL review skips all downstream compute (the cost-saving gate eduly already had, minus the label cascade). We deliberately **drop the identity-pinned `ai-review/quality-gate` Check Run** (a native review can't be pinned to a specific App; a bot's `REQUEST_CHANGES` doesn't hard-block) and the **#3414 workaround** ‚Äî because a `pull_request` job status enters the PR rollup normally, no App-authored head-SHA check is needed. The `‚úì/‚úó` labels stay pure badges; the App token stays cosmetic (recognizable `mtm-bot[bot]` identity). Human approval + the gated required tests remain the merge gate.

## Global Constraints

- **Gate mechanism (job-dependency, not identity-pinned check):** no App-authored `ai-review/quality-gate` Check Run is created. The action **exposes `verdict`/`confidence`/`merge_risk`/`review_event` as job outputs**; consumers gate heavy CI by making a `review-gate` job read `needs.ai-review.outputs.verdict` and fail red unless it is `pass`, with heavy lanes `needs: [review-gate]`. Consuming rulesets **must not require** the old `ai-review/quality-gate` context ‚Äî if a repo previously required it (eduly does), that context **must be removed from the ruleset during cutover** (Task 4.3) or PRs deadlock on a check that never posts. The new required contexts are the ordinary `review-gate` / `‚úÖ CI Complete` job statuses.
- **Authoring identity (explicit contract):** if the caller passes `app-id` + `private-key`, the review + labels are authored **under that GitHub App's own slug** (`<app-slug>[bot]` ‚Äî `mtm-bot[bot]` only because that is eduly's App; the action hardcodes no App). If they are omitted, the action falls back to `GITHUB_TOKEN` and authors as **`github-actions[bot]`**.
- **Labels are output, never a trigger; gate on the job output instead:** the `‚úì/‚úó /ai-review` labels are a visible badge only. **No workflow keys off `on: pull_request: labeled`** ‚Äî heavy CI gates on the ai-review **job's `verdict` output** (`needs:` + `if:`), not the label. (This is *why* we avoid label-driven CI: GitHub does not cascade triggers from `GITHUB_TOKEN`-authored label events, and even an App-authored label is a fragile trigger ‚Äî a job dependency is deterministic and #3414-free.) The authoring identity is therefore **purely cosmetic**: eduly passes the mtm-bot App creds only so the review/labels read as `mtm-bot[bot]`; `github-actions[bot]`/`GITHUB_TOKEN` behaves identically and does not affect gating.
- **Injection rule (ADR-0010):** every attacker-influenceable event value (label, branch, SHA, PR number, title/body) reaches the shell only via a step-level `env:` binding referenced as `$VAR` ‚Äî **never** inline `${{ ‚Ä¶ }}` in a `run:` block.
- **Enforcement split:** Claude produces content + verdict JSON only. A deterministic `github-script` step maps the verdict ‚Üí review event + labels. Never let posting depend on model output shape.
- **Versioning & supply-chain pinning (D13 ‚Äî audit's primary best-practice fix):** consumers pin the **first-party** composites `EdulyCom/github-actions/<action>@main` (floating; `CODEOWNERS` offsets floating-main's weak supply-chain guarantee ‚Äî D8). **Third-party** actions used *inside* the composites and self-CI (`anthropics/claude-code-action`, `actions/github-script`, `actions/create-github-app-token`, `actions/checkout`, `rhysd/actionlint`, `zizmor`) are pinned to a **full 40-char commit SHA** with a trailing `# vX.Y.Z` comment; a `.github/dependabot.yml` (`package-ecosystem: github-actions`) raises bump PRs. GitHub hardening guidance requires SHA-pinning third-party actions ‚Äî doubly so for a **public** repo other orgs copy verbatim.
- **Least-privilege & runaway-cost hardening (D14 ‚Äî audit reconciliation):** the App token is minted with **per-permission scope** ‚Äî `create-github-app-token` inputs `permission-pull-requests: write` + `permission-issues: write` (labels use the issues API) ‚Äî instead of inheriting the App installation's full permission set; it is minted in the **same job** that authors the review (the token auto-revokes in the action's post step and cannot cross jobs); the ai-review head-SHA `actions/checkout` sets `persist-credentials: false`; every `claude-code-action` job sets `timeout-minutes` (runaway-cost guard); thin callers keep workflow-level `permissions:{contents: read, pull-requests: write}` (least privilege).
- **Public & generalized:** the repo is **public** and org-agnostic. No action ships an eduly-specific default ‚Äî eduly values (App ID/key, runner label, health URL, test command, staging URL) are **caller-supplied**; the action's own defaults are neutral (`runner-label: ubuntu-latest`, no health-url/test-command default). READMEs, ADR 0001, and `docs/consumer-integration.md` are written generically with **eduly as one example consumer**, not the subject. A `LICENSE` is a **deferred, blocking decision**: a public repo with no license is "all rights reserved," so external reuse cannot begin until a license is chosen (see D12) ‚Äî internal EdulyCom consumption is unaffected.
- **Process:** all changes via PR (no direct push to protected branches); **merge commits, not squash**. TypeScript/no-`any` for any committed scripts; prefer inline `github-script`. Docs land in the same PR as their change.
- **Secrets & variables (org-level, human-supplied):** every secret/var the actions consume is set **at the EdulyCom org level** (`gh secret set --org EdulyCom ‚Ä¶` / `gh variable set --org EdulyCom ‚Ä¶`, repo-visibility scoped to the consuming repos incl. `github-actions` itself so its `selftest.yml` can run). The **implementer never invents secret values**: at each point one is needed (Phase 1 selftest, Phase 4 eduly adoption) the implementer **pauses and asks the human for the value**, then sets it at org level. Org-level secrets: `ANTHROPIC_AUTH_TOKEN` (eduly gateway bearer), `MTM_BOT_APP_PRIVATE_KEY`. Org-level vars: `ANTHROPIC_BASE_URL` (eduly gateway URL), `MTM_BOT_APP_ID`, `RUNNER_LABEL`, `AI_REVIEW_LABEL` (+ model-ID overrides if a gateway aliases them). (`GITHUB_TOKEN` is auto-provided per repo; not set here.)
- **Model IDs** (repo-var overridable): context `claude-haiku-4-5-20251001`; small-diff review `claude-sonnet-5`; large-diff review `claude-opus-4-8` (account HAS Opus access). Small = ‚â§15 files AND ‚â§400 changed lines.
- **Anthropic endpoint ‚Äî caller-supplied, defaults to Anthropic:** `anthropic-base-url` is an **optional caller input**; when omitted the action leaves `ANTHROPIC_BASE_URL` unset so `claude-code-action` uses the **standard Anthropic endpoint** (no gateway baked in ‚Äî keeps the action generic per D12). When the caller passes it (eduly ‚Üí its **custom gateway**), the value is plumbed to **every** `claude-code-action` invocation ‚Äî Haiku context, Sonnet/Opus review, ai-qa triage ‚Äî together with the caller's auth: either `anthropic-api-key` (‚Üí `ANTHROPIC_API_KEY`/`x-api-key`) or `anthropic-auth-token` (‚Üí `ANTHROPIC_AUTH_TOKEN` bearer). Because a gateway may expose **aliased** model names, model IDs stay repo-var overridable (a gateway lacking Opus falls back to the Sonnet ID via the override).

## Locked decisions

- **D1** mtm-bot-server ‚Üí **delete entirely** (repo + deployment + webhook subscription). **Keep the mtm-bot GitHub App _registration_ + private key** ‚Äî retained **purely as a cosmetic authoring identity** so the review/labels read as `mtm-bot[bot]`; functionally `github-actions[bot]`/`GITHUB_TOKEN` is identical (nothing triggers on the labels ‚Äî eduly's `ci.yml` gates heavy CI on the ai-review job's `verdict` output, not the label). New ADR supersedes 0017.
- **D2** ai-dev/ai-review/ai-qa skills ‚Üí **delete**; **retire the entire `autopilot` plugin** and relocate the surviving `version` (`ai-tooling-version`) utility skill to a new standalone plugin. Drop ai-uat's auto-fix routing to `/autopilot` (ai-uat files issues only); remove the security plugin's `/ai-review` + `/ai-qa` hook nudges.
- **D3/D4** Target repo ‚Üí **new `EdulyCom/github-actions`**.
- **D5** `auto-assign` ‚Üí set **assignee = PR author** on opened/ready_for_review.
- **D6** Structure ‚Üí **composite actions + thin caller**.
- **D7** Rubric ‚Üí **vendored markdown** `ai-review/rubric.md` (single source of truth after skill deletion). `/requesting-code-review` (superpowers) still installed live in CI.
- **D8** Versioning ‚Üí **floating `@main`** + CODEOWNERS.
- **D9** Opus ‚Üí **yes**; large diffs use `claude-opus-4-8`.
- **D10** ai-qa ‚Üí **post-merge non-blocking signal** (comment + labels) on push to default branch.
- **D11** Gate mechanism ‚Üí **CI-gating via ordinary job status** (the action exposes a `verdict` job output; a `review-gate` job fails red unless `verdict == 'pass'`; heavy lanes `needs: [review-gate]`), **not** an App-authored identity-pinned Check Run. No #3414 (the job runs on `pull_request`, so its status enters the PR rollup). Human approval + the gated required tests remain the merge gate. (Supersedes the earlier "advisory, CI runs regardless" framing: the review DOES gate CI ‚Äî for cost savings ‚Äî but via a job dependency, not a label cascade or an identity-pinned check.)
- **D12** Publish/distribution ‚Üí **public monorepo, path-referenced** (keep D6's one-dir-per-action layout; consumed as `EdulyCom/github-actions/<action>@<ref>`; **no** GitHub Marketplace listing ‚Äî leave that as a documented future extraction). Versioning stays **floating `@main` + CODEOWNERS** (D8; no semver tags) **for the first-party composites**; third-party actions inside them are SHA-pinned (D13). All defaults generic (eduly values in the caller). `LICENSE` **deferred**: publish the repo, but external reuse waits on the license choice; document this in the README and ADR 0001.
- **D13** Third-party action pinning ‚Üí **SHA-pin + Dependabot** (audit's primary fix). Every third-party `uses:` inside the composites/self-CI pinned to a full 40-char commit SHA + `# vX.Y.Z`; `.github/dependabot.yml` bumps them. First-party composites stay `@main` (D8) ‚Äî orthogonal: internal repo, CODEOWNERS-guarded.
- **D14** Least-privilege hardening ‚Üí App token **scoped per-permission** (`permission-pull-requests`/`permission-issues: write`), minted in the authoring job; `persist-credentials: false` on the head-SHA checkout; `timeout-minutes` on Claude jobs; **zizmor** GitHub-Actions static analysis added alongside `actionlint` in self-CI.

## Reference appendix (for the implementer)

- **Rubric source:** `~/.claude/plugins/cache/ai-tooling/autopilot/6.0.0/skills/ai-review/SKILL.md`.
- **Verdict math:** `confidence = 100 ‚àí 30¬∑P0 ‚àí 15¬∑P1 ‚àí 5¬∑P2` (clamp 0‚Äì100; test-quality adj ‚àí10 fail / ‚àí5 <60% crit / ‚àí15 no-tests). PASS iff `conf‚â•confidence-threshold ‚àß P0=0 ‚àß P1=0`, where `confidence-threshold` is the caller-supplied input (fallback `90`); the `CI‚âÝfail` term applies only on a re-review where a completed required-check conclusion exists (normally CI is downstream of the review ‚Äî see Task 2.2). Merge risk: any P0 or >2 P1 ‚Üí HIGH; 1‚Äì2 P1 or conf<70 ‚Üí MEDIUM; else LOW.
- **Markers ‚Äî dropped as a contract:** the old `<!-- ai-review-skill:review-comment ‚Ä¶ -->` / `<!-- ai-qa-skill:report-comment ‚Ä¶ -->` HTML markers were read by `mtm-bot-server` (now deleted) and used to extract the verdict for gating (now a **job output**). Both consumers are gone, so **no marker contract is required**. At most, prepend a minimal hidden `<!-- ai-review -->` / `<!-- ai-qa -->` anchor purely so the action can find-and-update its own sticky comment (or use `claude-code-action`'s `use_sticky_comment`); the native review is superseded by author-based dismissal, not a marker.
- **Verdict schema (model output):** `{verdict:"pass"|"fail", confidence:0-100, merge_risk:"low"|"med"|"high", intent:"aligned"|"partial"|"deviated"|"skipped", counts:{p0,p1,p2,p3}, review_event:"APPROVE"|"REQUEST_CHANGES", comment_markdown:string}`. The deterministic publish step **recomputes** `verdict`/`confidence` from `counts` and emits the result.
- **Action outputs (the gate contract):** `ai-review/action.yml` declares `outputs: { verdict, confidence, merge_risk, review_event }`, each mapped `value: ${{ steps.publish.outputs.<name> }}`. Consumers gate heavy CI by wrapping the action in a job (`outputs: { verdict: ‚Ä¶ }`) and adding a `review-gate` job that `needs:` it and fails red unless `verdict == 'pass'`, with heavy lanes `needs: [review-gate]`. **This ‚Äî not the label ‚Äî is the gate.**
- **Publish skeleton (github-script, author token, `env:{PR_NUMBER}`):** `pulls.createReview({...repo, pull_number, event, body})` ‚Üí `issues.addLabels({labels:[win]})` ‚Üí `issues.removeLabel({name:lose}).catch(e=>{if(![404,422].includes(e.status))throw e})` ‚Üí `core.setOutput('verdict'|'confidence'|'merge_risk'|'review_event', ‚Ä¶)`. **No `checks.create`.**
- **Authoring identity (cosmetic):** with App creds the identity is `<app-slug>[bot]` (eduly: `mtm-bot[bot]`); without them it is `github-actions[bot]` via `GITHUB_TOKEN`. The choice is **purely cosmetic ‚Äî the bot's display name.** Nothing triggers on these labels: heavy CI gates on the ai-review **job's `verdict` output**, never `on: pull_request: labeled`. (GitHub doesn't cascade triggers from `GITHUB_TOKEN` label events anyway ‚Äî which is *why* label-driven CI is avoided, not a reason to prefer the App.) eduly keeps the App only so review/labels read as `mtm-bot[bot]`.
- **claude-code-action inputs:** `prompt, claude_args, anthropic_api_key, claude_code_oauth_token, github_token, settings, plugins, plugin_marketplaces, additional_permissions, track_progress, use_sticky_comment`. Model via `claude_args:--model`; tools `--allowedTools`; structured output `--json-schema`. (standalone `model`/`allowed_tools`/`max_turns` DEPRECATED.)
- **Custom-gateway env for claude-code-action:** the underlying Claude Code CLI reads `ANTHROPIC_BASE_URL` and the auth env from the step's `env:` ‚Äî set `ANTHROPIC_BASE_URL` on every stage **only when the caller provides it** (omit ‚Üí standard Anthropic endpoint); supply **either** `ANTHROPIC_API_KEY` (x-api-key) **or** `ANTHROPIC_AUTH_TOKEN` (bearer), never both. **eduly's gateway ‚Üí bearer `ANTHROPIC_AUTH_TOKEN`** (caller passes `anthropic-auth-token`). Confirm during the Phase-2 spike that `claude-code-action` forwards these env vars (it invokes the CLI, which honours them) and that the gateway's model aliases match the routed IDs; if the action strips env, fall back to `settings`/a wrapper that exports them before the CLI runs.
- **Supply-chain & least-privilege (audit reconciliation ‚Äî D13/D14):** SHA-pin every third-party `uses:` (`anthropics/claude-code-action`, `actions/github-script`, `actions/create-github-app-token`, `actions/checkout`, `rhysd/actionlint`, `zizmor`) to a 40-char commit SHA + `# vX.Y.Z`; `.github/dependabot.yml` (`package-ecosystem: github-actions`, weekly) raises bump PRs.
